1) Recommended architecture (opinionated, buildable)
This is designed to be fast to ship, safe under concurrency, and integration-friendly.
1.1 Tech stack
Frontend: Next.js (React) + TypeScript + Tailwind
Auth/RBAC: Firebase Auth + Custom Claims (roles) + org membership records
Database: Firestore (multi-tenant, multi-location)
Backend: Firebase Cloud Functions (TypeScript) + Cloud Tasks (for sync jobs)
Payments: Stripe (subscriptions + payment intents)
Email: SendGrid (or Mailgun)
SMS: Twilio
Calendar: Google Calendar API + Microsoft Graph (Outlook)
E-sign: MVP “lightweight” (typed signature + timestamp + IP) + optional provider later
File storage: Firebase Storage (COI uploads, signed docs PDFs if generated)
Observability: Sentry + structured logs
1.2 Key design choice: “App is source of truth”
The app owns availability and conflict rules.
Google/Outlook are synced as downstream calendars (ops + invites), not authoritative.
2) Core data model (Firestore schema)
Multi-tenant, multi-location. “Org” can be your operating company; “Location” is each site.

2.2 Resource modeling (critical)
Each resource has:
type: "MODE" or "SEAT" or "ADDON"
exclusiveGroupId: e.g., location-wide (all modes share), seats have their own but also reference the group
capacity: attendees max
buffers: { beforeMins, afterMins }
approvalPolicy: { required: boolean, rules... }
requiredFields: array (attendeeCount, eventType, etc.)
requiredDocs: array (waiver, rental agreement, COI)
pricingProfileId: references price rules
Mutual exclusion rule:
MODE bookings lock the exclusiveGroupId for that location.
SEAT bookings lock the seat resourceId and (optionally) respect MODE lock via group.
3) Concurrency-safe conflict prevention (must-have)
Firestore can’t reliably do “start < end && end > start” with one query in all cases. Don’t gamble. Use slot-locking.
3.1 Slot locking approach
Normalize time into 15-minute slots (configurable)
Booking request expands to bufferStart / bufferEnd
Compute slot keys: YYYYMMDD-HHMM for each slot in range
In a transaction, create lock docs under:
orgs/{orgId}/locations/{locationId}/locks/{lockKey}/slots/{slotKey}
Where:
lockKey =
exclusiveGroup:{exclusiveGroupId} for MODE bookings
seat:{resourceId} for seat bookings
plus any additional locks you need (inventory-limited add-ons, etc.)
If any slot doc already exists → conflict → reject.
3.2 Lock release
On cancel, delete lock docs (transaction / batched deletes)
On booking end, optionally keep for audit; but typically delete to reduce storage.
4) RBAC + permissions (Firebase Auth + Firestore rules)
4.1 Roles
ADMIN
STAFF
MEMBER
PUBLIC
Role storage
Firebase Auth custom claims: { roleGlobal?: '...' } (optional)
Org membership record: orgs/{orgId}/users/{uid} includes:
role (ADMIN/STAFF/MEMBER)
status (active/suspended)
memberPlanId (if applicable)
companyId (optional)
Public users exist in Firebase Auth but may not have org membership until first booking.
4.2 Rule concept
Public can create a booking only through callable function (server enforced).
Members/staff/admin can read what they’re allowed to read.
Admin/staff can write operational data (holds, approvals, refunds, policies).
Implementation note: For integrity, all money-affecting actions (price calc, booking creation, refunds) should go through Cloud Functions, not direct client writes.
5) API contract (Cloud Functions)
Use Callable Functions for app actions + HTTP endpoints for webhooks/marketplace feeds.

6) Pricing engine spec (advanced pricing in MVP)
6.1 Price rules
Each location has priceRules that support:
base rate (public/member)
day-of-week modifiers
time-of-day peak/off-peak
weekend event uplift
demand modifiers (simple “occupancy threshold” uplift)
add-on prices and bundles
coupons/promo codes
gift card redemption
taxes + fees
deposits rules by resource type/attendee count
Important: The server generates a Quote Snapshot stored in booking.quoteSnapshot so the price can’t change after user starts checkout.

18) Definition of Done (MVP)
A build is “MVP complete” when:
No double booking is possible (locks + buffers + holds)
Member vs public horizons + pricing are enforced server-side
Public can book anything at public rates within the public window
Stripe subs + one-off + deposits + refunds work via webhooks safely
Required docs block confirmation when configured
Email + SMS reminders and key events fire reliably
Google + Outlook ops calendars receive updates; users get invites
Waitlist works with member-first notification window
Gift cards redeem correctly; inventory add-ons can’t be oversold (if enabled)
Access grants issue/revoke automatically (at least 1 provider)
Accounting + CRM sync jobs run and are retry-safe
Marketplace feeds publish availability; partner intake can create bookings (at least via feed)
Reports show utilization and revenue split; audit trail exists for changes and refunds
