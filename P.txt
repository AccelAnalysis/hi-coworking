# Product Requirements Document (PRD)

## Product name

**Hi Coworking Ops**

## Version

PRD v1.0 — January 27, 2026

## Owner

Jonathan / Accel Analysis (Product Owner)

## Problem statement

We need a single system to run a **micro co-working space** that is **reconfigurable** (coworking desks, podcast studio, live talks, conference setup, small meetings >10). The system must support:

* **Member priority booking + member pricing**
* **Public rentals for anything** at **public rates** on a **first-come, first-serve basis** (within a controlled public booking window)
* Strong **calendar management, scheduling, and user management**
* Operational controls (holds, approvals, buffers, check-in, contracts, reporting)

---

# 1) Goals and success metrics

## 1.1 Goals

1. Prevent double-bookings across **mutually exclusive space modes** and shared desks.
2. Support clear pricing and rules:

   * **Member pricing** vs **public pricing**
   * **Member priority** via longer booking horizon + optional waitlist priority
3. Enable fast self-serve booking for members and public, with automated ops workflows.
4. Provide an admin/staff control plane for daily operations and reporting.
5. Provide a growth path: convert public renters into members.

## 1.2 Success metrics (MVP)

* **0 double-booking incidents** after launch
* ≥ **80% bookings self-served** without staff intervention
* **< 2 minutes** average time to complete booking + payment for public users
* **> 90%** of bookings have correct pricing (member/public) and required documents signed
* Basic utilization and revenue reporting available weekly

# 3) Assumptions and policy decisions

These are “default” product rules unless changed later:

1. **Application is the source of truth** for availability. External calendars can receive events, but external events do not create bookings unless explicitly imported as blocks.
2. **Member priority is enforced primarily through booking horizons**, not bumping:

   * Members can book further in advance (e.g., 30 days)
   * Public can book closer-in only (e.g., 7 days)
3. Public rentals are FCFS **within the public booking horizon**.
4. The space can be in **one “mode” at a time** (mutual exclusion).
5. Coworking is typically booked by **seat (1–6)**, whereas podcast/talk/conference are booked by **exclusive mode**.
6. Some booking types may require **approval**, **deposit**, **waiver**, and/or **COI** based on admin-configured rules.

# 4) Users and personas

## 4.1 Personas

* **Member (Solo)**: wants reliable desk access, quick booking, member rates.
* **Member (Small Team)**: wants recurring meeting/podcast blocks, predictable billing, company account.
* **Public Renter (Creator)**: books podcast blocks, needs add-ons, may be new to the space.
* **Public Renter (Organizer)**: books live talk/small event, needs attendee capacity and terms.
* **Staff**: manages day-of operations, approvals, layout readiness, issues.
* **Admin/Owner**: configures pricing/rules, handles refunds, monitors utilization and revenue.

# 5) User journeys (core flows)

## 5.1 Public booking flow (FCFS + public pricing)

1. Choose resource (podcast / live talk / conference / seat if allowed publicly)
2. Select time slot (limited to public horizon)
3. Enter booking details (attendees, use case, add-ons, contact)
4. Accept required waivers/terms (and upload COI if required)
5. Pay (card; deposit if applicable)
6. Receive confirmation + calendar invite + instructions

**Key requirement:** Prevent public from booking beyond their allowed horizon.

## 5.2 Member booking flow (priority horizon + member pricing)

1. Login
2. Choose resource
3. View extended availability window
4. Book + apply member pricing
5. Confirmation + reminders

## 5.3 Staff day-of ops flow

* “Today view” timeline + arrivals + upcoming mode changes
* Approve/reject bookings
* Add holds/blocks
* Check-in management and incident logging

**Acceptance criteria**

# 7) Non-functional requirements

## Performance

* Availability search returns in **< 500ms** for typical ranges (7–30 days).
* Booking confirmation screen within **< 2 seconds** after payment success.

## Reliability

* Booking creation must be atomic with conflict checks (no race-condition double-booking).
* Webhook handling must be idempotent.

## Security & privacy

* RBAC enforced server-side.
* PII encrypted at rest where supported.
* Audit logs immutable (append-only).
* Rate limiting on public endpoints.

## Compliance

* Payment handled by Stripe; no raw card storage.
* Basic privacy policy + data retention policy placeholders (MVP).

# 12) Risks and mitigations

* **Double booking due to concurrency** → enforce server-side atomic booking creation with locking/transactions.
* **Calendar sync conflicts** → app as source of truth; import external only as holds.
* **Public misuse (events, damage)** → deposits, approval workflow for high-risk, COI upload rules, incident tracking.
* **Scope creep** → lock MVP to booking + pricing + horizons + payments + ops basics.
